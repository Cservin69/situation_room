## What this ships

### P3 — `authored_from` provenance signal (ADR 0014)

The fetch executor's `author_one` already takes one of two excerpt-construction
branches: real bytes from `prefetch_excerpt`'s `Some` path, or a synthesized
stub from `stub_excerpt` when the source has no `endpoint_hint`, the hint is
unparseable, or the pre-fetch returned an HTTP/transport error (the GDELT 429
case from the Session 20 live run). The branch was previously invisible to
anything downstream: a recipe authored against a stub looked indistinguishable
from one authored against real bytes.

This patch threads the branch through to the UI.

### P4 — Svelte 5 runes warning fix

`apps/desktop/src/components/dialogs/RecipeFlagDialog.svelte:101` previously read

```ts
const wasNonEmpty = initial.trim().length > 0;
```

which triggered Svelte 5's `state_referenced_locally` warning. `initial` is a
`$props()` value — reactive — and the eager read captured only the snapshot at
component mount. In current usage the dialog mounts fresh on each open, so the
snapshot was correct *in practice*, but the warning is load-bearing: a future
contributor changing the call site to keep the dialog alive across opens
would silently get a stale `wasNonEmpty`. The fix is `$derived(...)`.

Comment block on the new line names the trap explicitly so the next person
who reads the file finds the rationale rather than re-deriving it.

## Why option 3 (silent self-healing) is deferred, not shipped

The Session 21 handoff named three options for closing the provenance gap.
This patch ships options 1 + 2 (visible chip + dialog hint) and **defers**
option 3 (silent re-authoring of stub-authored recipes on the first
successful real-bytes fetch).

Option 3 is the architecturally cleanest, but it changes ADR 0007's
"runtime-is-LLM-free" invariant: today the runtime invokes the LLM only
on observable failure (the existing re-author-on-failure path); option 3
would add a second trigger on an *actively succeeding* recipe. That's an
ADR-0007-amendment-shaped decision, not a drop-in patch. ADR 0014
§"What the user does NOT see (option 3, deferred)" enumerates the four
questions a future amendment must answer (when does it trigger, how does
the user opt out, version vs overwrite, what does the UI show) and ties
the unlock to operational data we don't have yet.

The visible chip is the user's hook for triggering the manual path
themselves: they see `STUB-AUTHORED`, decide whether to re-run fetch,
and the existing failure-driven re-author path takes over from there.

## Why P1 still isn't shipped

P1 (live xAI verification of v1.8 against three slots — HTML-equivalent
USGS MCS, CELEX re-run, BAKED PDF source) is operator-machine work. The
sandbox this patch was built in has no network egress. The Session 21
handoff anticipated this: "the operator has a network-enabled machine;
this is the work that was blocked from the Session 20 sandbox."

The handoff also called out a cost-saving variant — re-running fetch
on plan `019de792-…` (the south-Korea-election plan from the live
run), since its recipes are already authored and persisted, costs zero
LLM tokens. **Now relevant in a new way:** with this patch landed, that
re-run will newly stamp existing recipes with `authored_from`. If the
GDELT recipe was authored from a stub (which it was — that's the
motivating case), the chip will appear on it after the next fetch run
*even without re-authoring*. Wait — actually no, it won't: existing
recipes load as `Unknown` (NULL on disk → Unknown in Rust); they only
get a populated `authored_from` if they're re-authored. So:

- An existing pre-v10 recipe → `Unknown` → no chip (silent legacy state).
- A *new* recipe authored after v10 lands → stamped at authoring time
  → chip appears or doesn't, accurately.

If the operator wants the chip on the existing GDELT recipe, the path
is "flag the recipe" (which doesn't re-author by itself, per ADR 0013)
+ "run fetch again with GDELT now reachable" (which triggers re-author
on the existing flag). The result is a recipe authored from real bytes,
stamped `FetchedBytes`, with no chip. That's the system working as
designed.

## Why P2 still isn't shipped

P2 is the top-of-file deferral banner on
`apps/desktop/failure_cases/recipe_author/2026-05-01-eur-lex-celex-instance-naive-selector.md`.

The Session 21 handoff said: "First step is to confirm which [is the cause]
. If `.gitignore`'d, decide whether failure-case files should be committed
(probably yes — they're load-bearing context for future sessions) or kept
local-only with a different propagation mechanism. Either decision is small;
deciding without the operator's input would be guessing."

Findings from this session:

- The whole `apps/desktop/failure_cases/` tree is **still absent** from
  `clean_code.zip` (same as Session 20).
- `.gitignore` does **not** mention `failure_cases/` — it is therefore
  **not** ignored. The tree simply isn't committed.

That's a decision needing operator input: the file may exist locally but
hasn't been committed, or it may be a working-only artifact the operator
maintains outside git. Either disposition is fine; this session declines
to pick for you. Two next-step options to flag:

1. **Commit the failure_cases tree.** Future sessions inherit the
   context. The Session 21 handoff said "probably yes." If you want this,
   a small follow-up: `git add apps/desktop/failure_cases/` and a
   one-line `.gitignore` audit to confirm nothing leaks (the directory
   shouldn't contain credentials or run artifacts).
2. **Keep failure_cases local, document the convention.** Add a short
   prose paragraph to `CONTRIBUTING.md` naming `apps/desktop/failure_cases/`
   as an operator-local research-artifact directory; remove the references
   to it from session handoffs, since they will keep frustrating sandbox-
   only sessions.

Either way, P2's banner work resumes the session after the directory
becomes reachable.

## Hard rules honored

- **xAI API key:** never read, written, or referenced anywhere in this patch.
- **ADR 0009 §"The rule":** no fresh `reqwest::Client::new()`. The patch
  adds no HTTP path; the executor's existing `SecureHttpClient` use is
  unchanged.
- **ADR 0007 runtime-is-LLM-free invariant:** unchanged. The new field
  records *what already happened during authoring*; the runtime branch
  on `prefetch_excerpt`'s success was already there. No new LLM call.
- **ADR 0011 plan lifecycle:** unchanged. No transitions added.
- **ADR 0013 recipe-feedback channel:** unchanged. The dialog gains an
  informational banner; the feedback channel itself (storage table,
  set/clear semantics, fenced-render security pattern) is untouched.
- **Closed extraction-mode enum:** unchanged. The signal is recipe-level,
  orthogonal to extraction.
- **No new dependency.** The patch is `std`-only on Rust and uses no
  new npm packages.
- **Generated TS files:** the hand-written files match ts-rs's existing
  output style. After the operator runs `cargo test -p situation_room-api`
  they regenerate; commit any byte-level diffs that surface (they should
  be cosmetic at most).
- **Migrations:** read 0008's comment block before writing 0010.
  Followed the same DuckDB-ALTER discipline.
- **Bounds checking on IPC string inputs:** unchanged. No new IPC strings
  introduced; `authored_from` is an enum, not a free-text field.
