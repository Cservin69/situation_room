## Summary

Operator-machine live verification run on 2026-05-03, against the
**post-Session-24 baseline** (Session 24 patch A applied — the BBC
RSS `endpoint_hint` is now in `config/sources.toml`). Topic:
"south-korea elections", plan
`019dee98-ae5c-7353-9835-9fe5449b988a`. Two sources nominated;
`rss_feeds` was one.

The Session 24 patch A's hint did its job at the authoring stage —
`authored_from = "fetched_bytes"`, the LLM saw real BBC News RSS
bytes during recipe authoring (good). But the recipe the LLM
authored was a regex over an XML structure assumption that BBC's
actual feed does not satisfy: the LLM's pattern wraps `<title>` in
`<![CDATA[…]]>`, which is a common RSS convention but **not**
BBC's. BBC News RSS emits unwrapped `<title>` content. The regex
matched nothing; the recipe failed at the Apply stage.

This is exactly the failure mode ADR 0012 §"Class B" defines —
"the LLM authored against … an idealized mental model of its
content, rather than the actual bytes at runtime." Even with
`fetched_bytes` provenance, the LLM's recipe encoded a
generalization (CDATA wrapping is "the standard RSS form") that
doesn't hold for the specific publisher.

## The failure message verbatim

```
extraction [regex_capture]: pattern matched nothing
```

(From `situation_room_pipeline::fetch_executor` warn log at
`2026-05-03T16:10:20.687268Z`.)

The error originates in `recipe_apply::extract_regex` at
`crates/pipeline/src/recipe_apply.rs`:

```rust
let caps = re.captures(text).ok_or_else(|| ApplyError::Extraction {
    mode: "regex_capture",
    reason: "pattern matched nothing".into(),
})?;
```

### Predicate-string discrepancy — useful evidence for ADR 0012

ADR 0012 §"Class B detection predicate" (line 264) names
`"matched 0 times"` as the RegexCapture predicate string:

```rust
message.contains("matched 0 times")           // RegexCapture
```

The actual runtime emits **`"pattern matched nothing"`** (verified
in `recipe_apply.rs::extract_regex`). The two are semantically
identical but byte-different. When (eventually) the deferred
automation lands, the predicate must use the runtime's real
string, not the ADR's anticipated one.

ADR 0012 explicitly anticipates this kind of correction —
"each string in this predicate must be verified against at least
2 observed live failures before being added. Do not add strings
speculatively." This case is observation #1 of the corrected
string. **Do not amend ADR 0012 today** — one observation of
`"pattern matched nothing"` does not meet the ≥ 2 bar; the
correction lands when a second observation arrives.

## Whether re-authoring succeeded, failed, or oscillated

**Pending.** Re-authoring was not attempted in this session.
ADR 0012's manual-practice protocol calls for the operator to:

1. Take the failure message, the recipe, and the actual fetched
   bytes (which the operator can capture with `curl` against the
   same URL).
2. Hand them back to the recipe author with the explicit instruction
   "the previous recipe's `pattern` assumed CDATA-wrapped titles;
   the source emits plain `<title>…</title>` instead. Re-author."
3. Observe whether the new recipe succeeds, fails the same way,
   fails differently, or oscillates between two near-identical
   wrong shapes.

The interesting expected outcome is *which* mode the LLM picks on
re-author. Options the closed extraction enum offers:

- A different `regex_capture` pattern: `<item>.*?<title>(.*?)</title>`
  (no CDATA brackets). Cheapest fix; survives most BBC feed
  variants.
- A `css_select`: `item title` against the XML-as-HTML parse.
  Works in `scraper`'s lenient mode (RSS is XHTML-shaped enough);
  more robust to whitespace and attribute variations.
- A `json_path`: not applicable — the source is XML, not JSON.

My speculation (not evidence): the LLM will pick the first option
because the change from the original pattern is minimal. Whether
that's the right call is an empirical question the operator can
settle.

### `rss_feeds` is a category descriptor — this evidence reflects on the category, not just BBC

The `rss_feeds` entry in `config/sources.toml` is documented as a
category of source rather than a specific publisher (Session 24
patch A header comment, lines 263–284 of `config/sources.toml`).
The BBC URL is just the structure anchor; the LLM is supposed to
pattern-match against it for any RSS feed. This case suggests
that pattern-matching from one RSS publisher's emission to
another's is fragile in ways the descriptor's prose acknowledged
("the inventions tend to be plausible-but-wrong") but the
recipe-author prompt does not currently teach the LLM to be
defensive about. Material for a future v1.5 prompt revision —
*not* for this session, per the Session 24 handoff §"What Session
24 is explicitly NOT" rule against prompt changes.
